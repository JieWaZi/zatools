# 写入安全参考

> 供所有可能写入 `wiki/`、修改结构关系或更新代码元数据的技能共享使用。

---

## 核心规则

中风险和高风险写入都必须在真正落盘前获得**显式确认**。

低风险、确定性的维护动作，在 workflow 已明确包含时可以直接执行。

---

## 风险分级

### 低风险

示例：

- 新建 document 镜像页
- 刷新索引或 graph
- 追加确定性日志
- 更新显然过期的派生输出

这类动作通常可逆，且不会改写知识归属。

### 中风险

示例：

- 把文档挂到已有 capability
- 追加辅助性 `code_refs`
- 把 change 关联到已有 capability
- 增加次级代码线索或 API 引用

这类动作会改变结构，但通常不重写系统中心。

### 高风险

示例：

- 新建 capability
- 合并或拆分 capability
- 新建或重分类 change
- 替换主 `code_refs`
- 修改 `change_classification`

这类动作会直接影响未来的检索、归属和规划行为，禁止静默自动写入。

---

## 确认协议

在中风险或高风险写入前，应当：

1. 展示提案
2. 区分事实与解释
3. 点明风险级别
4. 请求显式确认

好的确认表达类似：

- “这是一次中风险的 capability 关联追加，请确认后写入。”
- “这是一次高风险的 `new -> modify` 重分类，请确认后写入。”

---

## 何时分流到 `/devwiki-refresh` 或 `/devwiki-check`

以下情况更适合 `/devwiki-refresh`：

- wiki 与 raw 或代码发生漂移
- 路径、symbol、分类出现失配
- 用户在纠正旧错误

以下情况更适合 `/devwiki-check`：

- 需要确定性的健康扫描
- 用户想先看报告再决定修复
- 问题可能是坏链、`source_hash` 失配、反向链接缺失、`code_refs` 失效

---

## 提案最少内容

一个合格的写入提案应说明：

- 将要改什么
- 为什么要改
- 哪些证据支持这次修改
- 还有哪些地方不确定
- 用户确认后会发生什么

如果多个 capability 或 change 仍然都像候选，就不要自己拍板，应向用户提问。

---

## 破坏性操作

像 `/devwiki-reset` 这类破坏性操作要单独处理。

必须：

- 先做 dry-run
- 列出受影响的文件或 scope
- 涉及 `raw/` 时明确警告
- 等待显式确认后再执行

---

## 不该做什么

- 不要为了图方便，把高风险操作降级描述成低风险
- 不要把结构性改写藏在措辞调整里
- 不要静默改写归属关系或分类
- 不要把用户沉默当成确认
- 不要用 `/devwiki-check --fix` 去偷偷做 capability / change 重分类
