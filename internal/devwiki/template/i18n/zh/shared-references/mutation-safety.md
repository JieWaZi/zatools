# 变更安全约束

> 供任何会写入 `wiki/`、提出结构调整、或修改代码相关元数据的技能共享使用。

---

## 核心规则

中风险和高风险变更，写入前必须拿到**显式确认**。

低风险且确定性的维护动作，在 workflow 已经隐含授权时可以直接执行。

---

## 风险分层

### 低风险

示例：

- 刷新 `sources.hash`
- 刷新派生索引
- 追加确定性日志
- 更新明显过期的生成输出

这类动作通常可逆，不会重定义知识归属。

### 中风险

示例：

- 把 feature 挂到已有 capability
- 追加辅助性 `code_refs`
- 补充次级接口入口或测试入口
- 在不改范围的前提下收紧过时 feature 总结

这类动作会改结构，但通常不会重定义系统中心。

### 高风险

示例：

- 新建 capability
- 合并或拆分 capability
- 新建 feature
- 把 feature 改挂到另一个 capability
- 替换主 `code_refs`

这类动作会影响后续检索、归属和规划行为，绝不能静默自动写入。

---

## 确认协议

在任何中风险或高风险写入前：

1. 展示提案
2. 把事实与解释拆开
3. 明确标注风险级别
4. 向用户要显式确认

推荐确认话术：

- “这是一次中风险的 feature 挂载调整，请确认后写入。”
- “这是一次高风险的 capability 边界调整，请确认后写入。”

---

## 何时升级到 `/devwiki-refresh` 或 `/devwiki-check`

以下情况应走 `/devwiki-refresh`：

- wiki 知识与 raw 或代码漂移
- 路径、symbol 或 capability-feature 关系失配
- 用户在纠正之前写错的知识

以下情况应走 `/devwiki-check`：

- 需求是确定性健康校验
- 用户要先出一份报告
- 问题更像坏链、旧 hash、反向链接缺失或失效 `code_refs`

---

## 提案内容要求

一个合格的 mutation proposal 至少应包含：

- 要改什么
- 为什么这样改
- 证据是什么
- 还有什么不确定
- 如果用户确认，会发生什么

如果多个 capability 或 feature 仍然都像候选，就不要自己拍板，应向用户提问。

---

## 破坏性操作

像 `/devwiki-reset` 这种破坏性动作必须特殊处理。

始终要求：

- 先出 dry-run 计划
- 列出受影响文件或 scope
- 只要涉及 `raw/` 就要明确警告
- 等待显式确认后再执行

---

## 不该做什么

- 不要为了省事把高风险动作降级成低风险
- 不要把结构重写藏进一个小文案修改里
- 不要静默覆盖 capability 或 feature 的归属
- 不要把用户沉默当确认
- 不要用 `/devwiki-check --fix` 去偷偷做 capability 边界或 feature 归属重写
