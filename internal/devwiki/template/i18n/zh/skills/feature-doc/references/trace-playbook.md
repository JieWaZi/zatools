# 代码梳理步骤

## 总原则

- 先查资料，再查代码，再向用户提问。
- 必须追到业务闭环结束。
- 遇到通用基础设施只记录依赖，不继续深挖；如果基础设施里有该特性的专有逻辑，则继续往下追。

## 标准步骤

1. 明确特性名称与边界。
2. 在 `wiki/` 与 `raw/` 中检索同名能力、历史设计、历史复盘、代码总结。
3. 根据用户给的入口锚点进入代码排查。
4. 为每条链路记录：
   - 当前文件
   - 当前函数 / 类 / 路由
   - 上游调用者
   - 下游被调用者
   - 关键分支与条件
   - 关键数据结构 / 配置 / 表 / 外部接口
5. 完成一层后再进入下一层，直到形成完整业务闭环。
6. 交叉校验实际实现与现有文档是否一致。
7. 回填模板，整理“已确认事实”和“待确认事项”。

## 按入口锚点的排查顺序

### 用户给的是接口 URL

1. 找路由定义、中间件、控制器或 handler。
2. 找参数解析、权限判断、请求校验。
3. 找 service / use case / domain 层逻辑。
4. 找存储、缓存、消息、外部调用。
5. 找返回值组装与错误分支。

### 用户给的是关键文件

1. 通读文件，先标出公开入口。
2. 找入口的调用者与被调用者。
3. 沿着关键分支递进，不跳过 helper。
4. 补齐本文件外的上下游依赖。

### 用户给的是关键函数

1. 找定义。
2. 找全部调用方。
3. 找函数内部所有重要被调方。
4. 找输入来源、输出去向、副作用。

### 用户给的是页面路径 / 路由

1. 找前端路由与页面入口组件。
2. 找页面状态、表单、按钮动作。
3. 找 API 调用封装。
4. 继续按接口 URL 路径向后端追。

## 可以停止深挖的通用设施

- 通用日志
- 通用鉴权中间件
- ORM 基础封装
- 通用 HTTP 客户端
- 通用工具函数

前提是这些模块里没有该特性的专有逻辑。
