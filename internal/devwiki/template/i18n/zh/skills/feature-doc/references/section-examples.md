# 特性文档分节示例

下面的示例不是完整文档，而是告诉 agent 每一节应该写到什么粒度。

### 示例 1：核心特性小节

````md
### 2.1 用户组自动同步

- 能力说明：系统在用户组发生变更后自动同步关联成员与权限映射。
- 触发入口：`GroupSyncService#sync_group_members`
- 关键处理：加载目标用户组、比对期望成员列表、增量写入成员关系，并刷新权限缓存。
- 输出 / 副作用：更新组成员关系表、刷新缓存、写入同步审计日志。
- 关键约束：禁用状态的用户组不触发同步；手动锁定的成员记录不会被自动移除。
- 关键代码：`group_sync_service.rb#sync_group_members`
````

### 示例 2：模块设计小节

````md
### 5.1 GroupSyncService 模块

**模块职责**：负责执行用户组成员同步、权限刷新与同步结果落库。

**进程 / 入口信息**：

- 进程名：`app:group_sync_worker`
- 启动方式：后台任务触发
- 入口文件：`group_sync_service.rb`
- 入口函数：`sync_group_members`

**核心方法**：

|方法名|功能描述|关键设计|
|---|---|---|
|`load_target_members`|加载目标用户组及目标成员列表|分页读取、跳过禁用成员|
|`apply_member_delta`|增量更新成员关系并触发缓存刷新|批量写入、失败回滚、审计记录|

**配置 / 数据目录 / 外部依赖**：

- 配置文件：`config/group_sync.yml`
- 数据目录：`tmp/group-sync`
- 外部接口：`PermissionCacheClient#refresh_group_scope`

**关键实现细节**：

1. 先比对目标成员快照和当前关系表，再生成增删改计划。
2. 同步和缓存刷新放在同一事务边界附近执行，避免关系与缓存不一致。
3. 每次同步都写入审计事件，便于排查误同步或重复同步。
````

### 示例 4：规格与可靠性

````md
### 7.2 性能 / 时序规格

|指标|默认值|说明|
|---|---|---|
|同步批次大小|200 条|每轮批量处理的成员数量上限|
|重试最大次数|3 次|缓存刷新失败时的最大重试次数|

### 8.1 可靠性设计

- 去重或幂等：使用同步任务 ID 防止同一批次被重复执行。
- 重试与退避：缓存刷新失败后按 1s, 2s, 3s 线性退避。
- 持久化 / 备份：同步结果和审计记录都落库，支持后续重放与排查。
````
