# 特性文档分节示例

下面的示例不是完整文档，而是告诉 agent 每一节应该写到什么粒度。

### 示例 1：业务场景总览

````md
### 1.3 业务场景总览

|场景|触发方式|核心对象 / 数据|结果 / 影响范围|
|---|---|---|---|
|用户创建|管理员提交新建用户表单|用户基本资料、所属组织、初始角色|生成用户记录、建立组织归属关系，并使用户可登录|
|用户禁用|管理员执行禁用操作|用户状态、现有登录会话|更新用户状态、阻止后续登录并清理会话|
````

### 示例 2：核心特性小节

````md
### 2.1 用户创建

- 能力说明：系统允许管理员为组织创建新用户，并初始化其基础身份信息。
- 业务触发：管理员在后台提交“新建用户”表单时触发。
- 核心对象 / 数据：用户基本资料、所属组织、初始角色、登录凭证。
- 处理结果 / 影响范围：生成用户记录，建立组织归属关系，并使该用户可登录系统。
- 关键处理：系统先校验必填字段和唯一性，再写入用户记录，分配默认角色，并触发初始化通知。
- 关键约束：用户名和邮箱必须唯一；禁用组织下不允许创建用户。
- 辅助实现：`UserService#create_user`
````

### 示例 3：模块设计小节

````md
### 5.1 UserService 模块

**模块职责**：负责用户创建、更新、禁用和查询等基础能力。

**进程 / 入口信息**：

- 进程名：`web`
- 启动方式：HTTP 请求触发
- 入口文件：`user_service.rb`
- 入口函数：`create_user`

**设计目的**：把用户生命周期操作集中到服务层，避免控制器直接处理复杂业务逻辑。

**核心方法**：

|方法名|功能描述|关键设计|
|---|---|---|
|`create_user`|创建用户并初始化默认状态|先校验唯一性，再落库，再分配默认角色|
|`disable_user`|禁用用户并阻止后续登录|更新状态并清理登录会话|

**配置 / 数据目录 / 外部依赖**：

- 配置文件：`config/user.yml`
- 数据目录：不适用
- 外部接口：通知服务、审计服务

**关键实现细节**：

1. 用户创建前必须先校验用户名和邮箱唯一性。
2. 用户落库与默认角色分配应保持一致，避免出现“已创建但无角色”的中间状态。
3. 用户禁用时需要同步清理现有登录会话，确保禁用立即生效。
````

### 示例 4：API 接口

````md
### 6.1 API 接口

|接口名称|URL|方法|功能描述|
|---|---|---|---|
|`create_user`|`/api/users`|`POST`|创建新用户并返回用户标识|

**请求格式**：

```json
{
  "username": "alice",
  "email": "alice@example.com",
  "organization_id": "org-001",
  "role_ids": ["role-admin"],
  "operator": "admin@example.com"
}
```

**返回格式**：

```json
{
  "status": "ok",
  "user_id": "user-1001"
}
```
````

### 示例 7：规格与可靠性

````md
### 7.2 性能 / 时序规格

|指标|默认值|说明|
|---|---|---|
|创建接口超时|`5s`|单次创建用户请求的超时上限|
|重试最大次数|3 次|通知或审计写入失败时的最大重试次数|

### 8.1 可靠性设计

- 去重或幂等：以用户名或邮箱唯一约束防止重复创建。
- 重试与退避：通知发送失败后按 1s, 2s, 3s 线性退避。
- 持久化 / 备份：用户记录和审计事件都落库，便于后续排查。
````
