# DevWiki Feature 编写模板 v4（中文统一版）

> 适用位置：`wiki/features/<slug>.md`  
> 定位：Feature 是“功能契约 / 功能知识归档页”，不是详细设计原文，也不是代码实现说明。  
> 目标：用清晰文字沉淀功能目标、核心行为、关键规则、边界、联动和验收关注点，让后续 Agent 能准确回答“这个功能是什么、什么时候触发、有哪些规则、和什么有关、去哪里看实现”。

---

## 一、Feature 的职责

Feature 页应该回答：

- 这个功能解决什么问题？
- 对用户、产品、测试、运维而言，这个功能表现为什么？
- 功能什么时候触发？
- 关键规则是什么？
- 重要配置、状态、角色、边界是什么？
- 与其他功能有什么联动？
- 测试和验收要关注什么？
- 如果要看实现，应该进入哪个 workflow？

Feature 页不负责回答：

- 代码入口在哪里；
- 哪个类、模块、函数实现；
- 具体调用链怎么走；
- 状态具体在哪个变量或文件读写；
- 某个分支的代码如何修改；
- 详细排障命令和修复步骤。

这些内容应放到：

- `wiki/workflows/<slug>.md`：工程入口、调用链、关键实现、代码引用、修改影响；
- `wiki/troubleshooting/<slug>.md`：故障现象、日志、诊断路径、修复建议；
- `raw/`：原始设计文档全文。

---

## 二、Feature 与 Workflow 的边界

一句话区分：

```text
Feature = 功能行为和规则是什么
Workflow = 实现路径和代码怎么走
```

| 内容 | Feature | Workflow |
|---|---|---|
| 功能目标 | 详细写 | 可引用 |
| 用户场景 | 详细写 | 不展开 |
| 功能触发条件 | 详细写 | 可映射到代码 |
| 关键业务规则 | 详细写 | 说明在哪实现 |
| 配置对行为的影响 | 详细写 | 说明在哪读取/下发 |
| 状态/角色含义 | 功能级解释 | 说明代码如何判断 |
| 决策表/策略矩阵 | 保留功能级规则 | 展开实现分支 |
| 调用链 | 不写 | 详细写 |
| 文件路径/函数名 | 不写 | 详细写 |
| 修改影响 | 只写功能影响 | 写代码影响 |
| 排障步骤 | 只链接 | troubleshooting 详细写 |

---

## 三、写作原则

### 3.1 不要过度摘要

不能只写一句“该功能用于……”。应写清：

- 谁参与；
- 触发条件；
- 关键动作；
- 恢复或结束条件；
- 哪些场景不处理；
- 哪些配置影响行为。

### 3.2 不要复刻原始文档

不要把原始需求/设计文档的所有章节、接口字段、数据库字段、线程细节、代码结构都搬进 Feature。

Feature 应该提炼成“功能级知识”。

### 3.3 保留关键规则

如果某条规则会影响以下任一场景，就应该进入 Feature：

- 产品解释；
- 测试用例设计；
- 开发改动前理解影响；
- Agent 回答功能问题；
- 运维判断功能行为；
- 后续 workflow 代码定位。

### 3.4 实现细节只保留入口链接

Feature 可以写：

```text
实现定位见：[[<workflow-slug>]]
```

不要写具体文件、函数、handler、调用链。这些放到 Workflow。

```markdown

    kind: design
    hash: ""

    confidence: medium
    notes: ""

  - "<英文/代码/配置关键词>"

## 功能行为

- 与相关功能的联动。
```

## 六、设计信号抽取

生成 Feature 前，Agent 应先从原始资料中抽取设计信号：

```markdown
## 设计信号

### 功能目标

### 核心行为

### 关键规则

### 关键概念 / 状态 / 角色

### 关键配置

### 关键流程 / 流转

### 关键交互

### 可靠性 / 恢复 / 防护

### 观测 / 告警

### 边界 / 约束

### 验收关注点

### 需要放到 Workflow 的实现细节

### 需要放到 Troubleshooting 的排障细节

### 待确认问题
```

设计信号的作用是防遗漏，不代表 Feature 必须生成同名小节。

## 七、保留策略

### 7.1 必须进入 Feature

以下内容如果存在，通常应进入 Feature：

- 直接决定功能行为的规则；
- 用户或测试能观察到的状态变化；
- 影响开关、配置、联动、兼容的规则；
- 决定是否告警、恢复、隔离、重试、失败的规则；
- 重要异常场景和边界条件；
- 会影响后续代码修改风险的功能约束；
- 与其他功能的关键耦合关系。

### 7.2 放到 Workflow

以下内容默认放 Workflow：

- 代码入口；
- 调用链；
- 类、模块、函数；
- handler；
- 线程实现；
- 具体状态读写位置；
- 具体代码分支；
- 实现与设计的差异核对；
- 修改影响文件和测试文件。

### 7.3 放到 Troubleshooting

以下内容默认放 Troubleshooting：

- 具体日志关键字；
- 错误码；
- 排查命令；
- 诊断路径；
- 修复步骤；
- 现场问题处理经验。

### 7.4 可以不写

以下内容一般不写进 Feature：

- 原始文档模板中的空章节；
- 与功能行为无关的环境描述；
- 纯截图说明；
- 细粒度字段清单，且不会影响功能行为；
- 过时或未确认的历史信息。

---

## 八、Feature 质量检查

落盘前逐项检查：

- 是否说明了功能目标和核心行为；
- 是否保留了影响后续问答的关键规则；
- 是否避免把详细设计原文一比一搬进来；
- 是否避免过度摘要导致规则缺失；
- 如果功能依赖状态/角色，是否解释了关键状态或角色；
- 如果功能存在关键决策规则，是否保留了功能级规则；
- 如果功能受配置影响，是否保留了关键配置与行为影响；
- 如果功能涉及异常/恢复/自愈/回退，是否保留了关键条件和动作；
- 如果功能有观测/告警，是否保留了触发与恢复/清理规则；
- 是否写清与其他功能的关系；
- 是否把实现细节交给 workflow；
- 是否每个关键事实都有来源；
- 是否没有在 `sources` 中写代码文件路径或 `kind: code`；
- 是否把不确定内容写入来源说明。
