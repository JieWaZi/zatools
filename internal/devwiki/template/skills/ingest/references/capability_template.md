# DevWiki Capability 编写模板

> 适用位置：`wiki/capabilities/<slug>.md`  
> 定位：Capability 是“能力边界 / 能力地图页”，不是功能详细说明，也不是代码实现说明。  
> 目标：说明系统具备什么能力、为什么需要该能力、能力覆盖哪些 Feature、边界在哪里、与其他能力如何协作，让 Agent 能先从能力层做范围收敛。

---

## 一、Capability 的职责

Capability 页应该回答：

- 这个系统能力是什么？
- 它解决哪一类业务或技术问题？
- 它对用户、产品、测试、运维或系统稳定性有什么价值？
- 它覆盖哪些 Feature？
- 它不负责什么？
- 它与哪些能力协作或存在依赖？
- 它下面的功能边界如何划分？
- 后续要理解具体功能时，应该进入哪些 Feature？

Capability 页不负责回答：

- 某个具体功能的完整规则；
- 详细状态机、决策表；
- 代码入口、调用链、函数名；
- 具体排障步骤；
- 详细接口字段；
- 某个 Feature 的完整设计细节。

这些内容应放到：

- `wiki/features/<slug>.md`：具体功能行为、关键规则、配置、边界和验收关注点；
- `wiki/workflows/<slug>.md`：工程入口、调用链、代码引用、修改影响；
- `wiki/troubleshooting/<slug>.md`：故障现象、日志、诊断路径、修复建议；
- `raw/`：原始设计文档全文。

---

## 二、Capability 与 Feature 的边界

一句话区分：

```text
Capability = 系统具备什么能力、能力边界是什么
Feature = 这个能力下的具体功能怎么工作
```

| 内容 | Capability | Feature |
|---|---|---|
| 能力定义 | 详细写 | 可引用 |
| 业务价值 | 详细写 | 简要说明 |
| 能力边界 | 详细写 | 只写本功能边界 |
| 覆盖功能 | 列表和摘要 | 详细写 |
| 功能触发条件 | 不展开 | 详细写 |
| 功能关键规则 | 不展开，只摘要 | 详细写 |
| 状态机/决策表 | 不写 | 按需保留 |
| 代码入口 | 不写 | 不写，交给 Workflow |
| 排障步骤 | 不写 | 只链接，交给 Troubleshooting |

## 三、写作原则

### 3.1 Capability 是上层地图

Capability 应该帮助 Agent 和人快速判断：

- 问题属于哪个能力域；
- 应该继续查哪些 Feature；
- 哪些 Feature 之间存在关系；
- 哪些内容不属于这个能力。

它不是功能细节页。

---

### 3.2 不要复制 Feature 内容

Capability 中可以列出 Feature，并给一句摘要。  
不要把 Feature 里的关键规则、配置表、状态机、流程图复制过来。

正确写法：

```markdown
- VIP 接管：负责在主备状态变化时维护业务访问入口，详细规则见 [[feature-vip-failover]]。
```

错误写法：

```markdown
完整复制 VIP 接管 Feature 中的触发条件、决策表、状态流转……
```

---

### 3.3 要写清能力边界

Capability 最重要的价值之一是边界收敛。

需要明确：

- 这个能力负责什么；
- 不负责什么；
- 与哪些能力协作；
- 哪些问题应该转到其他能力。

---

### 3.4 适合做查询入口

Agent 在回答复杂问题前，可以先读 Capability，判断：

- 用户问题命中哪个能力；
- 应该查哪个 Feature；
- 是否还要查 Workflow 或 Troubleshooting；
- 是否可能涉及多个能力联动。

---

## 四、推荐模板

```markdown

slug: "<capability-slug>"
status: draft
summary: "<一句话说明该能力解决的问题>"
features:
  - "<feature-slug>"
related_capabilities: []
sources:
  - path: "raw/designs/<source-file>.md"
    kind: design
    hash: ""
    title: ""
    confidence: medium
    notes: ""
visibility: internal
confidence: medium
last_verified_at: YYYY-MM-DD
search_terms:
  - "<中文关键词>"
  - "<英文/代码/配置关键词>"
---

# <能力名>

## 能力摘要

用 3 到 6 条说明能力整体：

- 该能力解决哪一类问题；
- 主要面向哪些场景；
- 覆盖哪些核心功能；
- 与哪些能力协作；
- 最重要的能力边界是什么。

建议包含：

- 业务背景；
- 技术背景；
- 用户、运维、产品或系统稳定性价值；
- 没有该能力可能带来的风险。

|---|---|---|

## 入口导航

当用户问不同类型问题时，应该进入哪些页面。

| 用户问题类型 | 优先查看 |
|---|---|
| 了解能力整体 | 当前 Capability |
| 了解某个具体功能 | `wiki/features/<slug>.md` |
| 查看实现路径 | `wiki/workflows/<slug>.md` |

从能力整体角度提炼验收方向。

- 能力是否覆盖核心场景；
- 能力下各 Feature 是否边界清晰；
- 能力之间是否存在冲突或重复；
- 能力关闭、异常或缺失时，对系统有什么影响。
```

## 六、能力信号抽取

生成 Capability 前，Agent 应先从原始资料和已有 Wiki 中抽取能力信号：

```markdown
## 能力信号

### 能力目标

### 业务价值

### 技术价值

### 覆盖功能

### 不覆盖的范围

### 相关能力

### 核心概念

### 典型场景

### 能力级约束

### 需要拆分的 Feature

### 待确认问题
```

能力信号的作用是防遗漏，不代表 Capability 必须生成同名小节。

## 七、保留策略

### 7.1 必须进入 Capability

以下内容如果存在，通常应进入 Capability：

- 能力定义；
- 能力价值；
- 能力边界；
- 覆盖 Feature 列表；
- 能力之间的依赖、协作或影响；
- 能力级约束；
- 用户问题的入口导航。

### 7.2 放到 Feature

以下内容默认放 Feature：

- 具体功能行为；
- 功能触发条件；
- 参数、配置和取值规则；
- 状态、角色和决策规则；
- 功能级异常和恢复；
- 功能验收点。

### 7.3 放到 Workflow

以下内容默认放 Workflow：

- 代码入口；
- 调用链；
- 代码路径、类、函数；
- 状态读写位置；
- 修改影响和测试引用。

### 7.4 放到 Troubleshooting

以下内容默认放 Troubleshooting：

- 故障现象；
- 日志和错误关键字；
- 排查命令；
- 修复步骤；
- 现场处理经验。

---

## 八、Capability 质量检查

落盘前逐项检查：

- 是否说明了能力目标和价值；
- 是否写清能力范围内和范围外；
- 是否列出覆盖 Feature；
- 是否没有复制 Feature 的完整功能规则；
- 是否说明与其他能力的关系；
- 是否提供入口导航；
- 是否避免代码路径、函数名和调用链；
- 是否每个关键事实都有来源；
- 是否把不确定内容写入来源说明；
- 是否小节标题统一使用中文。
