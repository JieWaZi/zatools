# 代码追踪纪律

> 供 `devwiki-query`、`devwiki-code-to-doc` 以及所有需要从入口锚点走到真实实现的技能共享使用。

> 仅当工作流已经确认“这次确实需要看代码”时才使用本规范；如果文档已经能回答问题，`devwiki-query` 应先给出文档结论，再把代码核对作为可选补充。

---

## 核心规则

不要只停在第一个“看起来像”的文件上。

一条合格的 DevWiki 代码追踪，应从明确的**入口锚点**出发，沿着相关调用链往下走，直到足以解释核心行为。

---

## 合格的入口锚点

代码追踪应至少从以下一类开始：

- 接口 URL
- 关键文件
- 关键函数
- 页面路由
- 明确的特性名称
- 已知 capability 或 feature 名称

如果这些都没有，就先向用户提问，不要让代码搜索无边界扩散。

---

## 标准追踪流程

### Step 1：规范化入口锚点

把用户输入先转成可搜索对象。

例如：

- 接口 URL → 控制器 / 路由 / handler 候选
- 关键函数 → 直接 symbol 定位
- 页面路由 → 前端路由 + 后端接口候选
- 特性名称 → feature / workflow 候选 + top-K 代码命中

### Step 2：召回 top-K 候选

先用 `qmd` 与本地代码搜索拿到 top-K 候选，再压缩到最像入口的文件。

召回不等于理解，不能停在这一步。

### Step 3：核对入口文件

至少确认：

- 文件确实参与该特性
- 目标 symbol 存在
- 这是入口层，而不是随机辅助函数

### Step 4：沿调用链继续走

继续往下追踪：

- 调用者
- 被调用者
- 关键分支逻辑
- 持久化或外部接口边界
- 重要副作用

如果行为跨越多层，就要跟到关键设计决策清晰为止。

### Step 5：在清晰边界停止

一条追踪足够扎实时，至少能回答：

- 请求从哪里进入
- 哪些模块负责主行为
- 哪些函数 / 类承担关键实现
- 副作用发生在哪里
- 还有哪些地方仍未确定

---

## 何时向用户提问

以下情况不该继续硬猜，而应向用户提问：

- 多个入口文件同样像
- 用户提到的接口根本找不到
- symbol 查找多次失败
- 调用链因为动态分发或路由不清而断裂
- 代码搜索返回大量散点

一次只问 1 到 3 个最关键问题。

---

## 输出要求

一个合格的追踪总结应包含：

- 使用了哪个入口锚点
- 核对后的 top-K 候选
- 主调用链
- 关键实现节点
- 尚未解决的缺口

如果最终写成文档，面对大文件应优先使用 `path + symbol`，并写入 workflow 或 troubleshooting；不要写入 capability 或 feature。

---

## 不该做什么

- 不要看了一个顶层函数就停止
- 不要把文件名相似当成证据
- 不要因为第一层看起来简单就跳过内部调用
- 不要在追踪不完整时虚构调用链
- 不要在明显模糊时拒绝向用户提问
