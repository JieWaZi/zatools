# DevWiki Topic 编写模板

> 适用位置：`wiki/topics/<slug>.md`
> 定位：Topic 是产品语义层主题页，合并能力边界和功能规则，不写代码实现细节。
> 目标：让 Agent 能先判断主题是否命中，再读取产品语义下的核心机制和规则，再按需跳转 Workflow。

## 写作原则

`card` 只用于判断是否命中；`core` 写高频产品结论；低频补充、边界误区和待确认内容进入 `explain`；低频、低价值、高体积原文保留 raw。

Topic 只回答：

- 这个主题是什么；
- 功能边界是什么；
- 产品语义下的核心机制是什么；
- 关键产品规则、用户可见配置和状态是什么；
- 哪些边界、误区和联动需要注意；
- 需要看实现时跳转哪个 Workflow。

Topic 不回答：

- 代码在哪里；
- 函数怎么走；
- API path、接口字段、数据库表和内部配置字段是什么；
- 怎么测试；
- UI 页面有哪些字段；
- 复杂函数内部怎么校验参数；
- 内部流程、内部状态读写或内部调度如何完成。

## Topic 粒度

父 Topic 与子 Topic 可以共存，但必须职责清楚：

- 父 Topic 写能力族边界、主对象语义和关联能力地图；
- 子 Topic 写独立用户意图、独立规则和与父能力的边界；
- 父 Topic 只用一句话链接子 Topic，不复制子 Topic 规则；
- 子 Topic 只用一句话说明父 Topic 边界，不复制父 Topic 背景；
- 小能力无法独立用产品语言解释时，不拆 Topic。

## 核心内容

### 主题摘要

优先用短列表说明；不要为了压缩数量丢失影响理解、规则判断或路由判断的关键内容：

- 该主题解决什么问题；
- 产品语义下的核心机制是什么；
- 最重要的规则、状态或边界是什么；
- 与哪些对象、父子 Topic 或相邻主题存在联动；
- 需要看实现时进入哪个 Workflow。

### 功能边界

只在存在真实相邻主题或容易误路由时编写；没有复杂边界时可省略。

优先写成 1 段或短列表；如果边界超过短列表也不要删除关键边界，应按重要性排序，并把低频展开移到 `explain`：

- 

写作要求：

- 只写影响理解或路由判断的边界；
- 范围外只写真实容易混淆的相邻主题；
- 不写“不要写代码、测试、UI、其他无关功能”这类泛化排除项；
- 不写完整操作清单、UI 字段清单、接口、代码和测试细节；
- 用产品机制表达替代菜单/按钮/页面操作描述。

### 核心机制

说明该主题如何发挥作用，重点写产品机制，不写实现函数、接口、表或内部动作。

优先写成短段落，覆盖必要机制；不要为了保持短篇幅删除影响规则判断的关键机制：

- 输入或配置如何影响功能行为；
- 功能命中后会产生什么结果；
- 与相关对象如何联动；
- 哪些开关或状态会改变行为。

如果没有独立机制可写，不要为了填模板扩写背景。

模板小节不是填空题。没有独立决策价值的小节可以省略；不要为了凑完整而写背景套话、泛化排除项或低频细节。

## explain 写作要求

允许写：

- 典型使用场景；
- 容易误解的边界；
- 低频但重要的规则；
- 与其他主题的联动说明；
- 来源、冲突、不确定和待确认问题。

禁止写：

- 验收关注点；
- 测试方法、测试位置、测试用例；
- UI 字段清单；
- 接口字段、数据库字段、代码路径；
- 参数校验细节；
- 完整产品帮助文案；
- 为了填模板而写的背景套话。

## 质量检查

- card 是否只包含命中意图、关键边界和关联入口，没有核心规则摘要。
- Topic 不写代码路径、文件名、函数名、类名、handler、API path、接口字段、数据库表名、数据库字段、内部配置字段和调用链。
- Topic 不写内部流程、内部状态读写、内部调度或其他实现层表达；确需表达时是否只写了用户可感知的行为或结果。
- Topic 不写测试设计、验收清单、UI 字段清单或接口字段清单。
- 实现相关内容只写关联 Workflow 链接和一句话入口说明，不写代码路径。
- 功能边界只写真实相邻主题，没有泛化排除项。
- 父子 Topic 是否职责清楚，没有互相复制核心规则。
- explain 只作为 core 的低频补充。
- 没有为了填模板强行写空内容或背景套话。
- 不复制 raw 原文。
- 关系字段使用字符串数组。
- `module` 是该 Topic 所属的稳定模块 slug，用于 graph 聚合展示；不要创建独立模块页面。
