# 知识经济学放置规则

DevWiki 文档不是越完整越好，而是把知识放在最适合 Agent 读取的位置，降低单次检索和阅读成本。

## 基本原则

```text
命中判断、低体积 -> card
高频主结论、中体积 -> core
低频补充、高体积 -> explain
低频、低价值、高体积 -> 不进入 Wiki，保留 raw
```

`raw/` 保存完整证据，`wiki/` 保存可维护、可检索、可验证的知识摘要。不要把大段接口原文、历史讨论、字段全集、日志全集或低价值流水信息搬进 Wiki。

## 判断标准

放置内容前先判断三件事：

- 读取频率：后续查询、编码、排障是否经常需要；
- 决策价值：是否能帮助判断命中、理解规则、定位实现或决定修改影响；
- 体积成本：是否会让常用视图变长、变噪或难以扫描。

放置决策以读取频率、决策价值和体积成本为准。

`card` 只放判断“是不是这个页面”的信息，不提前回答规则细节。`core` 放高频高价值的主结论。`explain` 放低频但仍有价值的背景、展开、冲突和来源说明。低频、低价值、高体积内容保留在 raw，不进入 Wiki 正文。

## View 职责

`card` 是路由卡，不是压缩版正文。它只保留命中判断和必要边界，不提前展开规则、流程、字段或背景。

`core` 是主答案视图，承载高频、稳定、能直接帮助理解或决策的结论。它应避免把低频细节、完整清单或长流程塞进常用读取路径。

`explain` 是补充视图，承载低频但仍有价值的展开、边界误区、来源差异、冲突和待确认内容。

`card` 建议控制在 3 到 6 行；超过 500 中文字通常需要把内容下沉到 `core` 或 `explain`。具体页面类型如何组织 `card/core/explain`，由对应 Topic / Workflow / Troubleshooting 模板决定。

## 写作约束

- 不要为了“完整”把低频大体积内容塞进 `card` 或 `core`。
- 不要把 raw 原文复制到 Wiki；Wiki 只保留可复用结论、摘要和来源。
- 如果某段信息只在极少数排查或审计场景使用，优先放 `explain` 或保留 raw。
- 如果一个内容无法帮助后续查询、编码、排障或维护决策，不进入 Wiki。
- 不要为了填模板写泛化排除项、背景套话或没有独立决策价值的内容；没有内容就省略小节。
- 边界说明只写真实容易混淆的对象，不写“不要写其他无关内容”这类通用禁区。
- 具体页面的小节结构和表格字段以对应 Topic / Workflow / Troubleshooting 模板为准。
